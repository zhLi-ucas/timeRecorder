## 五个最有意思的决策

### 1. ChoiceWheel snap 的两段式修复（workaround → 真修复）

**v1.2 中段（`5bd7874`）**：横向 ChoiceWheel 用 `LazyRow + contentPadding + snap`。理想中：sidePadding = (viewport - itemWidth) / 2，snap 到 item 0 应该让首项精确居中。实际：foundation 的 `rememberSnapFlingBehavior` 在大 contentPadding 下左边界会算错 snap 目标，永远吸不到 item 0，用户看到的是 item 1 被选中。自写 fling behavior（spline decay + offset-distance snap）也没救回——边界舍入的根因在 LazyList 内部。**临时解法**：在列表最前塞一个不可选的「列表头」item（L1=「一级分类」、L2=「二级分类」），让边界 bug 落在这个 dummy 上，真实首项天然可选中。代价是用户滚动时会看到列表头掠过中心位（但松手会被吸回真实首项）。

**v1.2 发布后（`fff7e9d`）**：彻底抛弃 LazyRow，改写为 `Row + horizontalScroll(ScrollState)`。两端 `sidePadding` Spacer 让 item 0 物理上能滚到 viewport 中心；`pointerInput + awaitEachGesture` 监听手势翻 `userScrollPending` flag；松手时 `snapshotFlow { isScrollInProgress }` 收到滚动停止 → `animateScrollTo` 吸附 `scrollState.value / itemWidthPx` 最近项。`headerLabel` 保留作视觉提示但不再 load-bearing。这条把 v1.2 「已知遗留」里的「ChoiceWheel header 是 workaround」彻底关闭，是这一轮最重要的架构修正。

### 2. AI 上传 snapshot 包含双周期 + 完整 entries

用户决定：上传聚合 breakdown + 完整 entries（含 note/effectiveness），并且当前周期 + 上一周期 + 上次复盘一起发，让 AI 看到连续性。`AiReviewRequestBuilder.buildSnapshot` 直接复用 `StatsViewModel.computeStats` 的效能加权算法（valid/invalid 分钟拆分），保证 AI 看到的「浪费多少」与 STATS 屏口径一致。

Prompt 用用户原文 + 加「必须 JSON 返回 {summary, findings, adjust}」硬约束，解析失败 toast 不覆盖旧字段。

### 3. Room v2 → v3 严格走显式 Migration

v1.1 借 `fallbackToDestructiveMigration` 是用户授权的一次性破例。v1.2 加 `effectiveness` 列时严格写 `MIGRATION_2_3`：`ALTER TABLE time_entries ADD COLUMN effectiveness INTEGER NOT NULL DEFAULT 80`，再把无效消耗类的旧 entry `UPDATE ... SET effectiveness = 20`。老用户数据完整保留，新装用户走 seeder 默认值。

### 4. effectiveness 既影响 STATS 又上传给 AI

字段加在 `TimeEntryEntity` 上（0–100 Int，默认 80，无效消耗类默认 20）。三处消费：
- RECORD 屏 `EffectivenessRow` slider 录入
- STATS 屏 `computeStats` 加权聚合（valid/invalid 分钟）
- AI snapshot 同样用 valid/invalid 分钟，与 STATS 口径一致

单字段三处复用，避免数据碎片化。

### 5. AI 调用全链路安全网

- `withContext(Dispatchers.IO)` 不阻塞主线程
- OkHttp connectTimeout=15s / readTimeout=60s（DeepSeek 国内访问偶有抖动）
- 任何 IOException / JSONException / 非 2xx 都映射 `Result.Error(message)`
- 失败时 toast 显示 + **旧字段保留不覆盖**（用户已写的内容不会丢）
- 首次调用必须 AlertDialog 隐私确认，确认后 `ai_confirmed=true` 永久放行
- 无 API key 时按钮可点但 toast 直接报「未配置 API key」
- 模型可在 Settings 子页切（flash / pro）

## 协作模式总结

v1.2 沿用 v1.1 的 phased 实施模式（plan doc → 多轮对齐 → Phase A/B/C 分批实施 → 每步 build+install+smoke test → 绿后即时 commit）。Plan doc 落盘在前（`docs/v1.2-plan-2026-06-26.md`），实施在后，每 § 的「待确认问题」用户一次性答完后才动代码。Phase C 引入 plan mode 走了一次正式的 ExitPlanMode 流程，把 AI 集成的 14 个文件改动落成单个 plan 文件审核 + 实施。

ChoiceWheel 经历 3 轮用户反馈才稳定，每轮都是「装设备 → 用户测 → 报问题 → 改一稿」。最后一轮的「列表头」解法是用户主动提的——这种边界 bug 自己想可能要花更久。
