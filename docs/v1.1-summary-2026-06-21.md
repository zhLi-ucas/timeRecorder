## 关键技术决策

### 1. 中间 tab 用常驻 FAB（不是 NavigationBar 选中态）

Material3 `NavigationBar` 不支持中间槽位视觉区分。手写 `HomeBottomBar`：

```
Surface(height=80dp, tonalElevation=3dp) {
  Box {
    Row(align=BottomCenter, height=64dp) {      // 底栏
      NavItem[RECORD]  weight(1)
      NavItem[REVIEW]  weight(1)
      Spacer(width=64)                          // FAB slot
      NavItem[STATS]   weight(1)
      NavItem[SETTINGS] weight(1)
    }
    FloatingActionButton(                        // 浮起
      align=TopCenter, padding(top=4dp), size=56dp,
      containerColor=primaryContainer,
      contentColor=onPrimaryContainer
    )
  }
}
```

FAB 浮起 16dp，容器总高 80dp 避免裁剪；亮暗自适应配色走 `primaryContainer`。底栏在 RECORD 下**恢复显示**（v1 是隐藏的，因为 RECORD 现在是 tab 不是模态）。

### 2. 自定义时长用 `rememberSnapFlingBehavior`（不是 Modifier.scrollable）

`WheelPicker` 是手写 Compose 组件：

- `LazyColumn` + `rememberSnapFlingBehavior(lazyListState = listState)` 自动 snap
- `contentPadding = PaddingValues(vertical = itemHeight * halfVisible)` 让首尾 item 能滚到中央
- `snapshotFlow { listState.firstVisibleItemIndex }` 监听选中变化
- 5 行可见，选中行高亮（`onSurface`），±1 行 `alpha 0.7`，更远 `alpha 0.4`
- 上下两条 `HorizontalDivider` 框出选中区

`DurationPickerSheet` 装两个 wheel（小时 0–23 + 分钟 0/5/.../55）+ 顶部合计 + 底部确定/取消。修掉了用户报的"输入 15 视觉上被同步到 15m chip"的 bug —— 旧版 `OutlinedTextField` + chip 共享 `currentMin` 状态，新版彻底删掉文本输入，只剩 chip + wheel。

### 3. `BackHandler` 顶层兜底，不动 RECORD 屏内 `onCancel`

```kotlin
BackHandler(enabled = currentScreen != Screen.TODAY || settingsSubpage != SettingsSubpage.ROOT) {
    when {
        settingsSubpage != SettingsSubpage.ROOT -> settingsSubpage = SettingsSubpage.ROOT
        currentScreen != Screen.TODAY -> currentScreen = Screen.TODAY
    }
}
```

RECORD 屏本身的 `TopAppBar` 返回按钮（`onCancel`/`onDone`）和系统返回键**走的都是同一个 BackHandler**。Compose 嵌套优先级保证顶层 BackHandler 是兜底，RECORD 自己没注册 BackHandler，所以行为统一：从 RECORD 返回 → TODAY。

Android 13+ predictive back animation 在 targetSdk 36 下默认开启；Redmi/HyperOS 侧滑手势实测能正确触发 `BackHandler`。

### 4. Destructive migration（用户授权一次）

用户原话："旧的数据直接删除即可，都是测试数据"。Room schema 1 → 2 + `fallbackToDestructiveMigration`：

- 装新版后 Room 看到没有 1→2 的 Migration → drop 所有表 → 重建
- `DefaultDataSeeder.seedIfNeeded()` 看到 `KEY_SEEDED == null` → 用新列表 seed
- 验证：`adb exec-out run-as com.example.timemanager cat databases/timemanager.db[-wal,-shm]` → Python `sqlite3` 读取，确认 18 行（5 一级 + 13 二级），`cat_other.isSystem = 1`，旧的 `cat_core_work` 等已消失

**重要**：这是 v1.1 一次性破例。后续版本若再改 schema，必须写显式 Migration（已在 README 和 CLAUDE.md 标注）。

### 5. 新分类 seed 列表（5 一级 + 13 二级）

| 一级 | colorKey | 二级 |
|---|---|---|
| 工作 | blue | 实验 / 阅读 / 写论文 / 思考 |
| 休息 | cyan | 放空 / 游戏 / 阅读 |
| 自我实现 | orange | 规划 / 户外 / 室内 |
| 无效消耗 | grey | 无效等待 / 被动阅读 |
| 其他（系统） | neutral | 待分类 |

"阅读" 同时出现在工作和休息下，slug 用父前缀去重（`cat_work_reading` vs `cat_rest_reading`）。"其他" 替代原 "未分类"，保留 `isSystem = true`（不可删、不可归档）。
