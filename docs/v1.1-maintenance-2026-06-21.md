## 1. Tab 顺序调整 + 中间项视觉区分

### 现状

`MainActivity.kt:68-74` 的 `Screen` enum 顺序：TODAY / RECORD / STATS / REVIEW / SETTINGS。
默认进入 `Screen.TODAY`（`MainActivity.kt:80`）。底部 `NavigationBar` 按 enum 顺序展示。

### 目标

新顺序：**记一笔 / 复盘 / 今日账本 / 统计 / 设置**
中间位 `今日账本` 用常驻 FAB 风格圆按钮做视觉区分；默认进入仍为 `今日账本`。

### 方案

1. 调整 `Screen` enum 顺序为 `[RECORD, REVIEW, TODAY, STATS, SETTINGS]`。
2. `currentScreen` 默认值保持 `Screen.TODAY`。
3. 不用 Material3 `NavigationBar`，改用自定义 `BottomBar` Composable：

   ```
   Row {
     NavItem(RECORD)   weight 1f
     NavItem(REVIEW)   weight 1f
     FloatingActionButton(onClick = { currentScreen = TODAY })  ← 浮起圆
     NavItem(STATS)    weight 1f
     NavItem(SETTINGS) weight 1f
   }
   ```

   - FAB：直径 ~56dp，`offset = (-16).dp` 让它从底栏顶部凸出
   - 背景 `MaterialTheme.colorScheme.primaryContainer`，icon 用 `onPrimaryContainer`（亮色深蓝、暗色亮蓝）
   - 其他 4 个 NavItem 用经典 underline / color 选中态
   - TODAY 选中时：FAB 加外圈描边或缩放动画作反馈（可选）
4. RECORD 屏 `TopAppBar` 的返回按钮（`onCancel`/`onDone`）继续工作；底栏在 RECORD 下恢复显示（之前是隐藏的，本期改回可见）。

### 影响文件

- `app/src/main/java/com/example/timemanager/ui/activities/MainActivity.kt`：enum 顺序、自定义 BottomBar、移除 `NavigationBar`
- 新增 `app/src/main/java/com/example/timemanager/ui/components/HomeBottomBar.kt`：抽取自定义底栏

## 2. 侧滑回退 → 回主界面，不退出 app

### 现状

`MainActivity` 没有 `BackHandler`。侧滑 / 系统返回：
- 在 SETTINGS 子页（CATEGORIES/PROJECTS）→ 已有 `onBack`，正常
- 在任何一级 tab 根 → 系统默认 = 退出 app

### 目标（Q4 确认）

- 任何**非 TODAY** tab 侧滑/返回 → TODAY
- SETTINGS 子页 → SETTINGS 根（保持现状）
- TODAY → 系统默认（退出 app）
- RECORD 屏的 `TopAppBar` 返回按钮继续走 `onCancel`/`onDone`，不被影响

### 方案

在 `AppContent` 加顶层 `BackHandler`：

```kotlin
BackHandler(enabled = currentScreen != Screen.TODAY || settingsSubpage != SettingsSubpage.ROOT) {
    when {
        settingsSubpage != SettingsSubpage.ROOT -> settingsSubpage = SettingsSubpage.ROOT
        currentScreen != Screen.TODAY -> currentScreen = Screen.TODAY
    }
}
```

### 兼容性

- Android 13+ predictive back：`android:enableOnBackInvokedCallback="true"` on targetSdk 36 默认即开。
- Redmi 实测：侧滑手势 → `BackHandler` 触发，不依赖厂商适配。

### 影响文件

- `app/src/main/java/com/example/timemanager/ui/activities/MainActivity.kt`

## 3. 记一笔时长选择改造（30 / 60 / 自定义转动）

### 现状

`DurationInput.kt:24`：`QUICK_OPTIONS = listOf(15, 30, 60, 90, 120)`。
自定义是 `OutlinedTextField`，`onValueChange` 每次按键 `onMinChange(parsed)`。

**用户报的 bug**："想记录 150min，但写 15 时被同步到 15min 选项" —— 状态耦合：`currentMin=15` 命中 15min chip，chip 自动 selected，视觉混乱。最终值 150 实际是对的。

### 目标（Q2 确认）

只留 3 个 chip：**30m / 60m / 自定义**。自定义用双轮 wheel picker。

### 方案

1. `QUICK_OPTIONS = listOf(30, 60)`；加第 3 个 chip "自定义"，点击展开 `ModalBottomSheet`。
2. Sheet 内容：两个并排 `WheelPicker`：
   - 左轮：**小时** 0–23（共 24 项）
   - 右轮：**分钟** 0/5/10/15/20/25/30/35/40/45/50/55（共 12 项，5 分钟步进）
   - 选中行高亮 + 上下两条横线分割（iOS 风格）
   - 顶部显示当前合计：`3h 50m · 230m`
   - 底部"确定 / 取消"
3. `WheelPicker` 手写 Compose 组件（M3 无内置）。基于 `LazyColumn` + `rememberScrollState` + `animateScrollToItem` + `snapshotFlow` 监听中间 item。
4. 入口：选 30/60 → 直接写入 `currentMin`；选"自定义" → 打开 sheet，sheet 关闭时把 `H*60+M` 写入 `currentMin`。
5. 删除原"自定义文本输入框"。

### 影响文件

- `app/src/main/java/com/example/timemanager/ui/components/DurationInput.kt`：重写为 3-chip + sheet 入口
- 新增 `app/src/main/java/com/example/timemanager/ui/components/WheelPicker.kt`（约 100 行）
- 新增 `app/src/main/java/com/example/timemanager/ui/components/DurationPickerSheet.kt`：双轮 + 顶/底 UI

## 4. 分类：升级是否会覆盖 + 调整初始分类

### 现状（已查代码确认）

1. `DefaultDataSeeder.seedIfNeeded()`（`DefaultDataSeeder.kt:10-14`）由 `KEY_SEEDED` AppSetting 守护。**只在首启 seed 一次**，后续启动 / 升级 / 重装（保留数据）**不再 seed**。
2. `AppDatabase.kt:43-50` 没有 `fallbackToDestructiveMigration`。未来 schema 从 1 升 2 时**不写 Migration 会崩**，不会静默清数据。
3. 卸载 + 重装会清数据并重新 seed；OTA 升级不会。

**结论**：用户在设置页对分类做的修改（重命名、归档、新增）**在 app 更新后保留**，不会被覆盖。**只有"卸载重装 / 清数据"才会**回到默认。

### 目标

1. 在 plan.md / README 明示"分类自定义在升级后保留"。
2. 修订 seed 列表（影响新装 + 用户卸载重装；**当前 Redmi 上**用户需手动改或卸载重装才能吃到新默认）。

### 方案

1. 把 seed 列表与"升级不覆盖"写入文档。
2. 修改 `DefaultDataSeeder.kt` 的 `firstLevel` + `secondLevel`（等用户填完下方）。
3. 不做"恢复默认"按钮（避免误操作清掉用户的分类）；用户想拿新默认 → 卸载重装。
