下面是一份**尽量忠于柳比歇夫时间统计法**的手机 App 设计与本地 AI 执行计划。重点不是番茄钟、todo、AI 辅助，而是做一个**严肃的时间账本系统**。

# 柳比歇夫时间统计法 App 执行方案

## 0. 产品定位

本 App 的核心目标是：

> 让用户持续、低摩擦地记录每段时间实际花在哪里，并通过日、月、年统计，形成对自己时间结构、工作容量、任务耗时和长期积累的客观认识。

它不是：

* 不是番茄钟 App；
* 不是 todo list；
* 不是习惯打卡；
* 不是日记；
* 不是情绪记录；
* 不是 AI 助手。

它是：

> 时间账本 + 分类统计 + 周期复盘 + 计划校准工具。

最核心的对象不是“任务”，而是：

> 时间事件 Time Entry。

---

# 一、核心设计原则

## 1. 记录实际发生，而不是计划意图

柳比歇夫法的重点是**实际时间支出**。

所以 App 的主要记录对象应该是：

```text
某天某时到某时，用户实际做了什么，花了多久，归入哪个类别。
```

例如：

```text
2026-06-20
09:10-10:25
阅读《时间统计法》相关材料
类别：阅读 / 学习
用时：75 分钟
备注：整理基本原则
```

不要把 App 做成：

```text
今天我要完成 A、B、C。
```

而是做成：

```text
今天我实际把时间花在了 A、B、C、D 上，各花了多久。
```

---

## 2. 以统计为目的，而不是以提醒为目的

番茄钟的逻辑是：

> 现在开始专注 25 分钟。

柳比歇夫法的逻辑是：

> 今天结束后，我知道这一天真实流向了哪里。
> 一个月结束后，我知道这个月真实投入了什么。
> 一年结束后，我知道这一年真正积累了什么。

所以 App 的核心不是提醒、催促、倒计时，而是：

* 记录；
* 汇总；
* 分类；
* 对比；
* 回顾；
* 校准。

---

## 3. 宁可粗略完整，不要精细到放弃

忠于柳比歇夫法，不等于每分钟都完美记录。

App 应该鼓励：

```text
大致准确，长期连续。
```

而不是：

```text
极端精确，三天放弃。
```

所以必须支持：

* 实时计时；
* 事后补记；
* 批量补记；
* 未记录时间标记；
* 粗略时间段；
* 合并相似事件。

# 二、核心概念模型

整个 App 可以围绕 5 个核心对象设计。

## 1. 时间事件 TimeEntry

这是最重要的数据表。

每条记录表示一段实际发生的时间。

字段建议：

```text
id
date
start_time
end_time
duration_minutes
title
category_id
subcategory_id
project_id 可选
description 可选
result_note 可选
is_estimated 是否为估算
created_at
updated_at
```

其中最核心字段是：

```text
开始时间
结束时间
时长
事件名称
分类
```

不要让用户一开始必须填写太多东西。

最小记录可以只有：

```text
10:00-11:15
读书
75 分钟
```

---

## 2. 分类 Category

柳比歇夫法非常依赖分类统计，因此分类系统比 todo 更重要。

建议采用两级分类。

### 一级分类：时间账户

一级分类类似“会计科目”。

默认可以设置为：

```text
A. 核心工作
B. 辅助工作
C. 学习研究
D. 日常事务
E. 休息恢复
F. 社交沟通
G. 无效消耗
H. 未分类
```

### 二级分类：具体活动类型

例如：

```text
核心工作
- 写作
- 实验
- 编程
- 设计
- 推理/思考

辅助工作
- 资料整理
- 文件管理
- 环境配置
- 沟通协调

学习研究
- 阅读
- 听课
- 查资料
- 做笔记

日常事务
- 家务
- 通勤
- 行政处理
- 购物

休息恢复
- 睡眠
- 午休
- 散步
- 娱乐

无效消耗
- 无目标浏览
- 被动刷视频
- 无效等待
- 反复切换
```

注意：分类必须允许用户修改，但第一版要提供默认分类。

---

## 3. 项目 Project

项目不是必须字段，但建议保留。

分类回答的是：

> 这段时间属于哪种活动？

项目回答的是：

> 这段时间服务于哪个长期方向？

例如：

```text
类别：阅读
项目：某本书 / 某篇论文 / 某个研究方向
```

但为了忠于柳比歇夫法，项目系统不要做成复杂 todo。它只是统计维度。

字段：

```text
id
name
description
status: active / archived
created_at
updated_at
```

---

## 4. 周期汇总 PeriodSummary

汇总可以实时计算，也可以缓存。

需要支持：

```text
日汇总
周汇总
月汇总
年汇总
自定义区间汇总
```

统计维度包括：

```text
总记录时间
各一级分类时间
各二级分类时间
各项目时间
未记录时间
有效时间
无效时间
平均每日记录时间
最长连续时间事件
记录完整率
```

---

## 5. 复盘 Review

复盘不是日记，而是时间账本的解释层。

字段：

```text
id
period_type: day / week / month / year
period_start
period_end
summary_text
main_findings
adjustment_plan
created_at
updated_at
```

复盘内容应该简短，服务于统计解释。

例如每日复盘只问：

```text
今天主要时间花在哪里？
今天最有价值的时间段是什么？
今天最大的时间偏差是什么？
```

月度复盘问：

```text
本月核心工作时间是多少？
哪些类别占比过高？
哪些长期方向投入不足？
下月计划应如何修正？
```

## 页面 1：今日账本

这是首页。

目标：让用户看到今天的时间流向。

布局示例：

```text

08:30-09:00  早餐 / 日常事务 / 30m
09:00-10:20  阅读某资料 / 学习研究 / 80m
10:20-10:40  未记录 / 20m
10:40-12:00  写作 / 核心工作 / 80m
12:00-13:00  午饭休息 / 休息恢复 / 60m

今日已记录：4h 10m
未记录：20m
核心工作：1h 20m
学习研究：1h 20m

[开始记录] [补记时间] [整理未记录]
```

关键功能：

* 展示当天时间线；
* 自动发现时间空白；
* 支持点击编辑记录；
* 支持合并相邻记录；
* 支持把空白时间标为“休息”“未记录”“忽略”。

---

## 页面 2：记录

记录页负责快速创建 TimeEntry。

需要两种模式。

### 模式 A：实时计时

流程：

```text
点击开始
选择分类
填写事件标题，可选
开始计时
点击结束
补充备注，可选
保存
```

开始时只要求：

```text
分类
事件标题
```

结束时可选填写：

```text
结果备注
```

不要强制填写产出、评分、标签。

### 模式 B：事后补记

流程：

```text
选择开始时间
选择结束时间
填写事件标题
选择分类
保存
```

要特别重视事后补记，因为柳比歇夫法的现实可持续性很大程度依赖补记。

交互示例：

```text
补记一段时间

开始：14:00
结束：15:30
事件：整理资料
分类：辅助工作 / 资料整理
项目：可选
备注：可选

[保存]
```

不要只做“今天工作了几小时”，而要做分类账本。

# 六、关键算法

## 1. 时长计算

```text
duration_minutes = end_time - start_time
```

要求：

* 不允许结束时间早于开始时间；
* 支持跨天记录，但第一版可以先限制不跨天；
* 自动按分钟取整；
* 支持用户手动修正。

---

## 2. 时间冲突检测

当用户新增记录时，检查同一天是否已有重叠记录。

例如：

```text
已有：10:00-11:00
新增：10:30-11:30
```

应提示：

```text
该时间段与已有记录重叠。
选择：
1. 修改新增记录
2. 截断已有记录
3. 替换重叠部分
4. 仍然保存
```

MVP 可以先只做提示，不自动处理。

---

## 3. 空白时间检测

根据当天记录排序，找出未记录区间。

例如：

```text
08:00-09:00 已记录
09:00-10:30 空白
10:30-11:00 已记录
```

生成：

```text
09:00-10:30 未记录 90m
```

用户可以：

```text
补记
标为休息
忽略
```

---

## 4. 分类统计

按时间区间查询记录：

```sql
SELECT category_id, SUM(duration_minutes)
FROM time_entries
WHERE date BETWEEN ? AND ?
GROUP BY category_id;
```

二级分类、项目统计同理。

---

## 5. 记录完整率

用户设置每日统计范围：

```text
统计开始：07:00
统计结束：24:00
```

假设统计范围为 17 小时，即 1020 分钟。

```text
完整率 = 已记录分钟数 / 应记录分钟数
```

如果用户把睡眠也记录进去，则可以改成全天 24 小时制。

# 七、柳比歇夫法专属功能

为了区别普通时间记录 App，建议加入以下几个忠于方法的功能。

## 1. 时间账户

把分类命名为“时间账户”。

类似财务账本：

```text
核心工作账户
学习账户
辅助工作账户
休息账户
无效消耗账户
```

统计页显示：

```text
本月时间账户
```

而不是简单叫“分类”。

这样能强化“时间会计”的感觉。

---

## 2. 月度时间资产负债表

月报中显示：

```text
本月总记录时间：210h
核心工作：62h
学习研究：41h
辅助工作：38h
日常事务：30h
休息恢复：32h
无效消耗：7h
未记录：25h
```

并提供：

```text
本月核心工作占比
本月无效消耗占比
本月未记录占比
```

这比普通图表更贴近柳比歇夫法。

---

## 3. 年度累计表

柳比歇夫法的价值在长期累计。

所以必须有年度页。

展示：

```text
2026 年累计

核心工作：430h
学习研究：280h
辅助工作：190h
写作：120h
阅读：210h
编程：160h
无效消耗：75h
```

并支持查看趋势：

```text
1 月核心工作：45h
2 月核心工作：52h
3 月核心工作：39h
...
```

---

## 4. 任务耗时估算校准

这部分可以做得很简单，不要变成 todo。

设计一个“时间预算”功能。

用户可以在某个事件开始前填写：

```text
预计用时：60 分钟
```

结束后记录：

```text
实际用时：85 分钟
偏差：+25 分钟
```

长期统计：

```text
阅读类任务平均低估：1.4 倍
写作类任务平均低估：1.8 倍
配置类任务平均低估：2.3 倍
```

这是柳比歇夫法非常重要的收益：**校准自己对时间的估计能力**。

# 十二、最终判断

这个 App 的第一版不要追求“智能”，而要追求三个字：

> 账本感。

只要它能做到：

```text
记录方便
分类清楚
统计可靠
复盘连续
数据可导出
```

它就已经非常接近柳比歇夫时间统计法了。

后续再加 AI、todo、番茄钟都可以，但第一版必须先把“时间统计法”的骨架做扎实。
