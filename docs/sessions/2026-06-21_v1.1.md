## 关键技术决策

### 1. 中间 tab 用常驻 FAB（不是 NavigationBar 选中态）

Material3 `NavigationBar` 不支持中间槽位视觉区分，必须自定义。`HomeBottomBar` 用 `Surface(tonalElevation = 3.dp)` + `Box`：
- 底部 64dp 的 `Row` 装 4 个 `NavItem`，中间留 64dp `Spacer` 给 FAB
- FAB 56dp 用 `Alignment.TopCenter + padding(top = 4.dp)`，浮起 16dp
- 容器总高 80dp，FAB 不被裁剪
- FAB 配色 `primaryContainer / onPrimaryContainer`（亮暗自适应）

底栏在 RECORD 下**恢复显示**（v1 是隐藏的）—— 因为 RECORD 现在是 tab 不是模态。

### 2. 自定义时长用 `rememberSnapFlingBehavior`（不是 Modifier.scrollable）

`WheelPicker` 是手写 Compose 组件：
- `LazyColumn` + `rememberSnapFlingBehavior(lazyListState = listState)` 自动 snap
- `contentPadding = PaddingValues(vertical = itemHeight * halfVisible)` 让首尾 item 能滚到中央
- `snapshotFlow { listState.firstVisibleItemIndex }` 监听选中变化
- 5 行可见，选中行高亮（`onSurface`），±1 行 `alpha 0.7`，更远 `alpha 0.4`
- 上下两条 `HorizontalDivider` 框出选中区

`DurationPickerSheet` 装两个 wheel（小时 0–23 + 分钟 0/5/.../55）+ 顶部合计 + 底部确定/取消。

### 3. `BackHandler` 顶层兜底，不动 RECORD 屏内 `onCancel`

RECORD 屏本身的 `TopAppBar` 返回按钮（`onCancel`/`onDone`）和系统返回键**走的都是 BackHandler**。Compose 嵌套优先级保证顶层 `BackHandler` 是兜底，RECORD 自己没有注册 BackHandler，所以行为统一：从 RECORD 返回 → TODAY。

### 4. Destructive migration（用户授权）

用户原话："旧的数据直接删除即可，都是测试数据"。所以：

- Room schema 从 1 → 2
- `Room.databaseBuilder(...).fallbackToDestructiveMigration()` 一次清掉所有表重建
- `DefaultDataSeeder` 看到 `KEY_SEEDED == null` 自动重新 seed 新列表
- 验证：adb pull DB → sqlite3 dump → 看到 5 个新一级（cat_work / cat_rest / cat_self / cat_invalid / cat_other）+ 13 个新二级，旧的 cat_core_work 等已不存在，`cat_other.isSystem = 1`

后续版本若再改 schema，应写显式 Migration 而不是继续 destructive。
