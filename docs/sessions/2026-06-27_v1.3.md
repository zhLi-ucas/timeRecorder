## 关键技术决策

### 1. Dense Pager 替代 Int.MAX_VALUE Pager

TODAY 屏用 `pageCount = Int.MAX_VALUE` + pivot，靠"钳制未来日"实现"只能看过去"。STATS 借鉴但不同——空时段必须**直接跳过**不留空页（用户原话："中间缺失了记录，则统计会直接跳过"）。

方案：提前算出所有"有数据的 anchor 列表"，`pageCount = anchors.size`。空时段根本不存在页，跳过天然成立。代价是 VM 多一份 `validAnchors: StateFlow<List<LocalDate>>`，加 `observeAllDates()` DAO。换来用户体感"滑一格就是下一个有数据者"。

### 2. anchor 记忆 vs 下标记忆

用户首次提问就敲定要"按 anchor 记忆"：用户在 STATS 看 6/8 那周，期间通过 RECORD 加今天 entry，列表前面会长出"本周"。如果按下标记忆（idx=2），用户会被动跳到 2 周前；按 anchor 记忆（记住"6/8 那周"），列表变化后 `idx = anchors.indexOf(target)`，用户继续看原时段。

实现：VM 内 `_userAnchor: MutableStateFlow<LocalDate?>`，validAnchors 每次 emit 时 init 块重算 idx；anchor 被删空则 clamp 到最新（`anchors.size - 1`）。

### 3. `{period}` 占位符 + JSON_FORMAT_SUFFIX 始终追加

prompt 让用户随便改，但格式约束不能被破坏。方案：
- 用户可编辑部分 `DEFAULT_DEEPSEEK_PROMPT` 含 `{period}` 占位符
- 发送前 `replace("{period}", "日"/"周"/"月")`
- `JSON_FORMAT_SUFFIX` **始终追加**在用户 prompt 末尾，防解析失败

用户即使删空 prompt，AI 仍能基于 JSON 后缀 + user content 推断出结构化输出。

### 4. MONTH 复盘用对话框替代数字配置

用户原话提议："月的频率不是很高，可以在月的 ai 总结时弹出选择框"。频率低（12 次/年）+ 用户控制权 + 透明度都赚到了。

实现：`generateWithAi()` 入口判断 MONTH 分支 → 异步 `buildMonthWeekOptions(today)` → 写到 `_monthPicker` StateFlow → UI 监听弹 `MonthReviewPickerDialog` → 用户确认后调 `confirmMonthPicker()` → `proceedAiCall(MONTH, today, selectedMondays)`。

DAY/WEEK 仍走设置中的数字（频率高，dialog 太烦）。

### 5. 预览功能的两处入口 + 真实数据

Settings 子页预览固定用 WEEK + today 真实数据（让用户调 prompt 时看到 `{period}` 替换效果 + 实际 JSON 结构）。REVIEW 屏预览用当前 period + form 的 periodStart 作为 anchor（用户实际生成前的最后一道 review）。

为了不重复 fetchRecentReviews 逻辑，Settings 子页用了内联的简化版 `fetchRecentReviewsPreview`（DAY/WEEK 走配置数字，MONTH 直接 emptyList——预览固定 WEEK 不进 MONTH 分支）。**已知小重复**，未来若加新预览入口可抽 helper。

### 6. Pager 状态同步：settledPage + LaunchedEffect(key)

`HorizontalPager` 与 VM 双向同步是个微妙的问题。两次 bug 修复最终方案：

- **pager → VM**：`LaunchedEffect(pagerState) { snapshotFlow { pagerState.settledPage }.collect { selectPage(it) } }`——用 `settledPage` 而非 `currentPage`，避免动画过程中发射中间页
- **VM → pager**：`LaunchedEffect(selectedIdx, anchors.size) { if (settledPage != selectedIdx) animateScrollToPage(selectedIdx) }`——直接用 key 触发，**不**用 `snapshotFlow { StateFlow.value }`（后者不跟踪 StateFlow）

## 关键 bug 修复历程

STATS 滑动从"有 bug"到"完全工作"经历了 3 轮用户反馈，是本期最耗时的部分：

1. **首轮**（`e414de1`）：用 `currentPage` + `snapshotFlow { StateFlow.value }` 标准模式。用户报"方向反 + TODAY 空白 + 滑动行为诡异"
2. **二轮**（`81ffbab` + `49268a7`）：先修方向（sortedDescending → sorted），再换 `settledPage` 解决动画争用。用户报"仍空白，但只在 TODAY"
3. **三轮**（`459f494`）：识别出 `snapshotFlow { StateFlow.value }` 不跟踪的根因，换成 `LaunchedEffect(key)`。终于完全工作

教训：Compose 的 `snapshotFlow` 是为跟踪 Snapshot state 设计的，**不要**用来跟踪 kotlinx.coroutines StateFlow。后者要用 `LaunchedEffect(stateFlow收集后的值)` 或 `flow.onEach { ... }.launchIn(scope)`。

## Post-release fix（同日稍晚，commit `f107639`）

用户实战测试后发现两个相关问题：

### Bug A：DatePicker focus 跳到昨天

**症状**：REVIEW tab 点右上角日历 icon，DatePicker 弹出后高亮的不是 `form.periodStart` 当天，而是前一天。导致用户选了 6/27 看着像选对了，实际 form 切到了 6/26——AI 生成上传的是 6/26 的数据但用户以为是 6/27。

**根因**：Material3 DatePicker 内部按 **UTC** 解释 millis。原代码：
```kotlin
initialSelectedDateMillis = form.periodStart
    .atStartOfDay(ZoneId.systemDefault())   // ← bug
    .toInstant().toEpochMilli()
```
在 UTC+8（中国）系统时区下，`2027-06-27 00:00 +08:00 = 2027-06-26 16:00 UTC`，转成 millis 后 DatePicker 看到的就是 06-26。读取路径 `atZone(ZoneId.systemDefault())` 同样错。

**修复**：两处 `ZoneId.systemDefault()` → `ZoneOffset.UTC`。`ReviewScreen.kt` + `TodayLedgerScreen.kt`（后者防御性同源修复，用户没报但同样的 bug）。顺手清掉未使用的 `import java.time.ZoneId`。

### Bug B：预览融入生成流程（二次确认）

**症状**：REVIEW 屏原来有独立的"预览上传内容"按钮 + "AI 生成草稿"按钮，两者分离。用户点生成后无法确认"上传的 JSON 真的对应我现在看的时段"，尤其 DatePicker 还有 Bug A 的偏移问题——疑虑叠加。

**修复**：
1. **删除 REVIEW 屏独立的"预览上传内容"按钮**——预览不再是单独动作
2. **`generateWithAi` 流程改造**：DAY/WEEK 不再直接 `proceedAiCall`，先调 `buildAndShowPendingPreview` 生成预览快照写到 `_pendingPreview: StateFlow<PendingPreviewData?>`；MONTH 流程的 `confirmMonthPicker` 选完周后也走预览中转
3. **UI 监听 `pendingPreview`**，非空时弹 `PreviewDialog`（带新增的 `onConfirm` 回调）；用户点"确认生成" → `confirmGenerate()` 从 `_pendingPreview` 取参数调 `proceedAiCall`；点"取消" → `cancelPendingPreview()` 不发请求
4. **关键不变量**：`proceedAiCall(period, anchor, selectedMondays)` 的三个参数全部来自 `PendingPreviewData`——即用户在 dialog 看到的预览内容对应的数据。"所见即所传"，根除配对疑虑

**PreviewDialog 扩展**：加可选 `onConfirm: (() -> Unit)?` + `confirmLabel` 参数。有 onConfirm 显示"确认生成/取消/复制全部"；无则保持"关闭/复制全部"（Settings 子页的预览按钮不变，仅看 prompt 效果）。

**ReviewViewModel 清理**：删除不再使用的 `_previewText` / `_previewLoading` StateFlow 和 `requestPreview()` / `clearPreview()` 方法。Settings 子页用的是 `SettingsViewModel` 自己的同名 state，不受影响。

### 教训

- **Material3 DatePicker 永远用 UTC**：`initialSelectedDateMillis` 和 `selectedDateMillis` 都是 UTC 时间戳。`atStartOfDay(ZoneOffset.UTC)` + `atZone(ZoneOffset.UTC)` 是唯一正确做法。系统时区转换会让选中日偏移一天（UTC+ 时区跳到昨天，UTC- 时区跳到明天）。
- **预览与生成共享数据快照**：当用户对"上传内容是否匹配当前视图"有疑虑时，最稳妥的设计是预览和生成消费同一份 `PendingPreviewData`——预览只是展示，确认时把同一份数据传给实际执行函数，杜绝"预览 A 生成 B"的可能。
