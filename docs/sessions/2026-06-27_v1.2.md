## 关键技术决策

### 1. ChoiceWheel 用「列表头」绕开 LazyRow 左边界 snap bug

LazyRow + 大 `contentPadding` + `rememberSnapFlingBehavior` 在左边界会算错 snap 目标位置（item 0 永远吸不到位，item 1 反而被选中）。自写 fling 也救不回——边界舍入逻辑根因在 foundation 内部。最终方案：在 LazyRow 最前塞一个**不可选**的 label item（「一级分类」/「二级分类」），让边界 bug 落在这个 dummy 上，真实首项天然可选中。snapshot flow + fling snap 都按 `headerOffset = 1` 偏移。代价是用户视觉上会看到列表头掠过中心位（但松手后会被吸回真实首项），相比修不了边界 bug 完全可接受。

### 2. AI snapshot 包含双周期 + effectiveness-weighted 分钟数

用户决定：上传**聚合 breakdown + 完整 entries**（含 note/effectiveness），且当前周期 + 上一周期 + 上次复盘三件套一起发，让 AI 能看到连续性。`AiReviewRequestBuilder` 复用了 `StatsViewModel.computeStats` 的效能加权算法（valid/invalid 分钟），保证 AI 看到的「浪费多少」与 STATS 屏一致。

### 3. Room v2 → v3 走显式 Migration（不再 destructive）

v1.1 借 `fallbackToDestructiveMigration` 是用户授权的一次性破例。v1.2 加 `effectiveness` 列时严格走 `MIGRATION_2_3`：`ALTER TABLE ... ADD COLUMN effectiveness INTEGER NOT NULL DEFAULT 80` + 把无效消耗类的旧 entry 修正为 20。老用户数据完整保留。

### 4. Settings 子页用现成模板，BackHandler 自动覆盖

`AiDeepSeekScreen` 复制了 `CategoryManagerScreen` 的 `Scaffold + TopAppBar + LazyColumn` 模板。`SettingsSubpage` enum 加 `AI_DEEPSEEK` 值，`MainActivity` 的 `when (settingsSubpage)` 加分支，**顶层 BackHandler 不动**——它只判断 `!= ROOT`，新子页天然接住返回。

### 5. AI 调用走 IO dispatcher + 60s timeout

`DeepSeekApi.chat` 用 `withContext(Dispatchers.IO)` 不阻塞主线程；OkHttp 客户端 connectTimeout=15s / readTimeout=60s；任何 IOException / JSONException / 非 2xx 都映射成 `Result.Error(message)`，UI 层 toast 显示后旧字段保留不覆盖。
