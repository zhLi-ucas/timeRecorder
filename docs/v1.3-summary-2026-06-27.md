## 五个最有意思的决策

### 1. Dense Pager 替代 Int.MAX_VALUE Pager

TODAY 屏的 `pageCount = Int.MAX_VALUE` + pivot + "钳制未来日"模式不适合 STATS——STATS 的硬约束是"中间缺失的时段直接跳过"，不能留空页。Dense Pager 提前算"有数据的 anchor 列表"，`pageCount = anchors.size`，空时段根本不存在页。换来"滑一格就是下一个有数据者"的干脆感。代价是 VM 多一份 `validAnchors: StateFlow` 和一个 `observeAllDates()` DAO。

### 2. anchor 记忆 vs 下标记忆

用户在 STATS 看 6/8 那周时通过 RECORD 加了今天 entry，`validAnchors` 前面会长出"本周"项。按下标记忆，用户被动跳到 2 周前；按 anchor 记忆，列表更新后 `idx = anchors.indexOf(target)`，继续看原时段。VM 内 `_userAnchor: MutableStateFlow<LocalDate?>`，validAnchors emit 时 init 块重算 idx；anchor 被删空则 clamp 到最新。

这条是用户首次提问就拍板的，避免了"被动跳页"的糟糕体感。

### 3. `{period}` 占位符 + JSON 格式始终追加

用户希望 prompt "有一句话能自动根据当前选择的日周月变化"。方案：默认 prompt 含 `{period}` 占位符，发送前 `replace("{period}", "日"/"周"/"月")`。用户可在任意位置放占位符或不放（不放就原样发送）。

关键约束：`JSON_FORMAT_SUFFIX`（`必须以 JSON 返回 {summary, findings, adjust} ...`）**始终追加**在用户 prompt 末尾——即使用户删空 prompt，AI 仍能基于 JSON 后缀推断结构化输出。这条保底让解析失败率几乎为零。

### 4. MONTH 复盘用对话框替代数字配置

用户原话提议："月的选择可以给用户自己...在月的 ai 总结时弹出选择框，里面包含了该月的涉及到的几个周，然后按用户自己的勾选情况上传对应周的周报即可"。

频率低（12 次/年）+ 用户控制权 + 透明度都赚到了。DAY/WEEK 频率高（365 / 52 次/年），仍走设置中的数字。

实现：`generateWithAi()` 入口判断 MONTH → 异步 buildMonthWeekOptions(today) → `_monthPicker` state → UI 监听弹 `MonthReviewPickerDialog`（默认勾选已存周，未存 disabled，"全选已存"快捷）→ 用户确认后 `proceedAiCall(MONTH, today, selectedMondays)`。

本月一个周报都没有时不弹 dialog，toast 引导去 REVIEW 写至少一周。

### 5. 预览功能的两处入口

Settings 子页预览：固定 WEEK + today 真实数据，让用户调 prompt 时看到 `{period}` 替换效果 + 实际 JSON 结构。用输入框 draft（不必先保存）。

REVIEW 屏预览：用当前 period + form 的 periodStart 作为 anchor，用户实际生成前的最后一道 review。

两处都通过 `previewText: StateFlow<String?>` 暴露，UI 监听弹 `PreviewDialog`（滚动 monospace + "复制全部"按钮）。

## 三次 STATS 滑动 bug 修复

最耗时的部分。从"有 bug"到"完全工作"经历了 3 轮用户反馈：

### Bug 1：滑动方向反 + REVIEW 没法选历史时段

首次实现用 `sortedDescending`（idx=0 最新，idx 大是过去）。但 TODAY 屏是 asc（idx 大是未来），所以同样的手势在 STATS/TODAY 产生相反语义。

修复：`sortedDescending` → `sorted()`，init 默认 idx 改 `anchors.size - 1`（最新）。同时给 REVIEW 屏加日期选择器（之前只能从历史列表点"打开"加载已有 review，没法对未存 review 的历史时段写新 review）。

### Bug 2：TODAY 切换后空白（用 settledPage 修）

`LaunchedEffect(pagerState) { snapshotFlow { pagerState.currentPage } }`——`currentPage` 在 `animateScrollToPage` 动画过程中会发射中间页。每次发射触发 `selectPage()` 把 `_selectedIdx` 改成中间值，与另一边的 `animateScrollToPage(idx)` 来回打架，最终 pager 卡在 `initialPage` 渲染空白 Spacer。

修复：`currentPage` → `settledPage`，只在滚动完全停止后变化。

### Bug 3：TODAY 仍空白（根因：snapshotFlow 不跟踪 StateFlow）

`LaunchedEffect(viewModel) { snapshotFlow { viewModel.selectedIdx.value } }`——这是关键陷阱。`snapshotFlow` 是为跟踪 Compose Snapshot state（如 `MutableState`）设计的，**`StateFlow.value` 不是 Snapshot state，不在跟踪范围**。所以这个 snapshotFlow 只发射一次初始值就再也不动了。

WEEK 工作是因为首次创建 pagerState 时 init 已跑完，`initialPage = 4`（WEEK latest）正好对；切到 TODAY 时 pagerState 已存在不重建，依赖动画把 page 从 4 推到 8——动画根本没触发，page 卡在 4。`anchors[4]` 在 TODAY 列表里是 06-23，与 VM 的 `anchorDate = 06-27` 不等，渲染 Spacer 空白。

修复：改用 `LaunchedEffect(selectedIdx, anchors.size) { if (settledPage != selectedIdx) animateScrollToPage(selectedIdx) }`。`selectedIdx` 来自 `collectAsState()` 是真正的 Compose state，跟踪可靠。

教训：**永远不要用 `snapshotFlow { stateFlow.value }` 跟踪 StateFlow**。要么 `LaunchedEffect(composeState)`，要么直接 `flow.onEach { ... }.launchIn(scope)`。

## 协作模式总结

v1.3 沿用 v1.2 的 phased 实施模式（plan doc → 多轮对齐 → Phase A/B 分批实施 → 每步 build+install+smoke test → 绿后即时 commit）。两 Phase 独立交付：

- **Phase A（§1 STATS 滑动 + 复盘区）**：先用 plan mode 走正式 ExitPlanMode 流程，plan 文件落盘到 `~/.claude/plans/partitioned-yawning-music.md`。5 个 Step 实施，1 次 build 通过。但实战测试发现 3 个 bug，分 3 次 commit 修复

STATS 滑动 bug 修复是本期最值得反思的部分——3 次修复暴露了 Compose `snapshotFlow` 跟踪语义的微妙之处。详细分析见 `docs/sessions/2026-06-27_v1.3.md`。
